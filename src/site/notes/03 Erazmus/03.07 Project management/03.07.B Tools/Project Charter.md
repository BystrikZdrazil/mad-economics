# Purpose
A project charter is a document issued by the project owner or sponsor that authorizes the project manager to use organizational resources for project activities

>[!quote]+
>A document made by the project owner or project sponsor which gives the project manager the power to use the company´s resources for the project´s activities

- Describes scope, goals, participants, roles, responsibilities, and authority
- In PMBOK it is the central document authorizing the project
- In PRINCE2 it corresponds to the project mandate → project brief → project initiation document
- In IPMA it is part of Practice 2: Requirements and objectives
- The project charter is usually short and may refer to other detailed documents

# Purpose
- Creates a shared understanding of the project among team members, management, project owner, and stakeholders
- Serves as a focal point throughout the project
- Acts as a baseline for team meetings and control meetings
- Clarifies reasons for undertaking the project
- Outlines objectives, constraints, and directions for the solution
- Identifies main stakeholders
- Defines in‑scope and out‑of‑scope items
- States target benefits, high‑level budget, and spending authority

# When to use the tool
- Prepared in the opportunity identification phase
- Created by the project owner or at their request
- May require preliminary analyses before finalization
- Once approved, it is not updated
- Functions as an up‑front contract for project initiation

# How to use the tool
- **Step 1: Collection of information**
    - Stakeholders are involved through meetings, interviews, brainstorming, workshops, brainwriting
    - People with relevant professional, organizational, or geographical experience contribute
    - Lessons learned from previous projects are used
    - Supported by the checklist for gathering information
- **Step 2: Conducting analyses and surveys**
    - Preliminary analyses may be needed before drafting the charter
- **Step 3: Develop project charter**
    - Document is prepared clearly to avoid later misunderstandings
    - Template is used to structure content

# Checklist for Gathering Information
- **Overall requirements specification (scope)**
    - What problems the project solves
    - What needs are covered
    - What is new
    - Which stakeholders benefit
    - What happens if the project is not completed
    - Alternative solutions
    - Whether results can be achieved by expanding ongoing activities
- **Business case**
    - Who decides project success
    - Crucial criteria
    - Approval responsibility
    - Measurable benefits and how to measure them
    - Responsible parties for achieving benefits
    - Expenses or losses caused by the project and how to measure them
    - Responsible parties for limiting losses
    - Stakeholders experiencing losses
- **Overall budget**
    - Project cost
    - Uncertainty in calculations
    - Cost of maintaining and running the result
    - Total investment description (e.g., cost‑benefit analysis)
- **Overall schedule**
    - Optimal project completion period
    - Time after which project is irrelevant
    - Time before which project is irrelevant
    - Expected cost period
    - Expected yield period
- **Project organization**
    - Project manager selection
    - Reporting structure
    - Project manager authority
    - Steering committee composition
    - Resource needs and availability
    - Subcontractor or partner needs and accessibility
- **Stakeholder analysis**
    - Whether a stakeholder analysis has been prepared
- **Risk analysis**
    - Main results of risk analysis
    - Project prerequisites
- **Quality level**
    - Defined quality parameters
    - Required quality management standards
    - Special tests needed
- **Subcontractors**
    - Which services/products to buy vs deliver internally
    - Agreements with key suppliers
- **Assumptions and dependencies**
    - Prerequisites for project progress
    - Dependencies on other projects
- **Strategic alignment**
    - Whether the project relates to company strategy

# Template Structure
- Executive summary
- Overall requirements specification
- Business case
- Overall budget
- Overall schedule
- Project organization
- Initial stakeholder analysis
- Risk analysis
- Quality plan
- Subcontractors
- Assumptions and dependencies
